\chapter{Implementation}
\label{chap:implementation}

This chapter details the practical realization of the \textit{Deterministic QoS over Wi-Fi} system model within the OMNeT++ 6.0.3 and INET 4.5.4 simulation frameworks. All implementation was carried out entirely in OMNeT++ with INET, ensuring modular integration with existing networking components. The implementation focuses on translating the abstract concepts of bandwidth reservation, coordinated listening windows, and emergency handling into functional modules at the application and MAC layers. The core components developed include custom traffic generators, a QoS-aware packet classifier, and a finite-state machine (FSM) integrated into the MAC layer for enforcing the prescribed channel access states.

\section{Application Layer: Traffic Generation}

To evaluate the proposed QoS mechanisms under various traffic conditions, two custom application modules were implemented. Both utilize Differentiated Services Code Point (DSCP) markings to signify packet priority, influencing their treatment further down the network stack.

\subsection{UdpCritApp: Configurable Priority Traffic}

The \texttt{UdpCritApp} module is designed to generate continuous streams of UDP traffic with configurable priority levels, simulating routine V2X messages (e.g., periodic sensor updates).

\textbf{Design and Features:}
\begin{itemize}
    \item \textbf{Criticality Levels}: Implements a four-level priority system (\texttt{LOW}, \texttt{MEDIUM}, \texttt{HIGH}, \texttt{CAR\_CRASH}).
    \item \textbf{DSCP Mapping}: Assigns standard DSCP values to each priority level: 8 (LOW), 16 (MEDIUM), 24 (HIGH), and 46 (CAR\_CRASH).
    \item \textbf{Configuration}: The priority level for an instance is set via the \texttt{criticality} parameter in the network configuration file (NED).
\end{itemize}

\textbf{Implementation Details:}
The module extends INET's \texttt{UdpBasicApp}. Its key modification is the override of the \texttt{setSocketOptions()} method. During initialization (\texttt{INITSTAGE\_LOCAL}), the socket is configured to mark the Type of Service (TOS) field for all outgoing packets with the DSCP value corresponding to its configured criticality level. This ensures network-level prioritization without altering the core packet generation logic of its parent class.

\subsection{UdpCollisionApp: Emergency Event Simulation}

The \texttt{UdpCollisionApp} module is tailored to simulate the bursty, high-priority traffic characteristic of emergency events in V2X scenarios, such as collision messages.

\textbf{Design and Features:}
\begin{itemize}
    \item \textbf{Fixed Criticality}: All generated packets are marked with the highest priority (DSCP 46, \texttt{CAR\_CRASH}).
    \item \textbf{Event-Based Bursts}: The start time of an emergency burst is randomized within a configurable interval (\texttt{collisionStartMin} to \texttt{collisionStartMax}). The burst itself has a fixed duration (\texttt{burstDuration}), during which packets are sent at a regular interval.
    \item \textbf{Dynamic Destination}: Supports sending bursts to multiple pre-configured destination addresses, randomly selecting a target for each packet within the burst to simulate broadcast or multi-cast scenarios.
\end{itemize}

\textbf{Implementation Details:}
This module inherits directly from \texttt{ApplicationBase} and implements the \texttt{UdpSocket::Callback} interface for fine-grained control over its lifecycle. It uses self-messages to manage its state: a timer for the randomized start (\texttt{START}), a periodic message for sending packets during the active burst (\texttt{SEND}), and a timer to stop transmission after the burst duration elapses (\texttt{STOP}). The DSCP value of 46 is hard-coded within its \texttt{setSocketOptions()} method.

\subsection{QoS-Aware Packet Classification}
\label{sec:qosclassifier}
A crucial supporting component is a packet classifier. This classifier inspects the DSCP field of incoming IP packets and maps them to the appropriate IEEE 802.11e Access Category (AC) queue within the MAC layer. This ensures that high-priority packets (e.g., DSCP 46) are placed into the high-priority \texttt{AC\_VO} queue, directly influencing their scheduling and transmission order according to the rules of the current FSM state. 

Figure~\ref{fig:classifier_flow} illustrates the packet classification flow. When a packet arrives, the classifier first checks whether it is IPv4 or IPv6. In the case of IPv4, the DSCP field is extracted directly, while for IPv6 the Traffic Class field is shifted to obtain the DSCP value. Each DSCP value is then mapped to the corresponding User Priority (UP). Non-IP packets are mapped by default to the best-effort class (\texttt{UP\_BE}). Finally, the packet is tagged with its computed User Priority and forwarded through the output gate. 

This mechanism guarantees consistent QoS-aware mapping across both IPv4 and IPv6 traffic, allowing differentiated handling of flows based on application requirements.

\subsection{Finite-State Machine Controller}

The core of the DQ-WiFi implementation is the \texttt{EdcaFsmController}, a novel submodule integrated into the Hybrid Coordination Function (HCF) of INET's \texttt{Ieee80211Mac} model. This controller acts as a central gatekeeper, intercepting events from both the upper layers (applications) and the lower layers (PHY) to enforce the state machine defined in the system model. It encapsulates all timing logic and state variables---\texttt{activePeriod}, \texttt{listenPeriod}, and \texttt{blockingEndTime}---and provides helper methods (\texttt{switchState()}, \texttt{isListening()}, \texttt{isBlocked()}) to the MAC layer for coordinated decision-making.

The integration was achieved by modifying key methods in the HCF/EDCA architecture to consult the FSM controller before proceeding with standard operations. The following figures and explanations detail this integration point-by-point.

\subsubsection{Integration with Upper-Layer Packet Processing}
The ingress path from the network layer is the first point of control. Figure~\ref{fig:upper_fsm} illustrates the modified \texttt{processUpperFrame()} method, where FSM logic now gates channel access requests.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\linewidth]{Figs/upperframe.png}
    \caption{FSM logic integrated into \texttt{processUpperFrame()}.}
    \label{fig:upper_fsm}
\end{figure}

\textbf{Implementation Details:}
\begin{enumerate}
    \item \textbf{Classification and Enqueuing:} Incoming IP packets are first processed by the QoS-aware classifier (Section~\ref{sec:qosclassifier}).
    \item \textbf{FSM Permission Check:} After enqueuing, the \texttt{EdcaFsmController} state is consulted before any request for channel access is made.
    \item \textbf{State-Driven Access Logic:}
    \begin{itemize}
        \item \textbf{ACTIVE State:} By default, all ACs contend normally under CSMA/CA. If \texttt{BEAperiodic} is disabled, behaviour is IEEE 802.11e-compliant. If enabled, AC\_BE traffic is silenced, reallocating bandwidth to mission-critical queues.
        \item \textbf{LISTENING State:} Only AC\_VO and AC\_BE may transmit. AC\_VO retains its low-latency parameters, while AC\_BE inherits VO’s tighter AIFSN and contention window, draining its queue rapidly inside a protected window.
        \item \textbf{Emergency Handling:} If the \texttt{carCrash} flag is active but the FSM is in \texttt{LISTENING}, normal channel requests are blocked, ensuring uninterrupted emergency exchanges.
        \item \textbf{General Access:} In other cases, the method checks \texttt{isBlocked()}. If not blocked, it calls \texttt{maybeRequestChannelAccess(ac)}, which follows EDCA contention but within the temporal contract enforced by the FSM.
    \end{itemize}
\end{enumerate}

The alternation between ACTIVE and LISTENING is governed by the configurable pair \texttt{activePeriod}/\texttt{listenPeriod}, so reshaping the bandwidth contract requires no code changes. This provides fine-grained enforcement, ensuring packets are only submitted during their authorized FSM states.


\subsubsection{Integration with Lower-Layer Frame Processing}
The egress path for outgoing and incoming frames is handled by \texttt{processLowerFrame()}, which now incorporates FSM checks as shown in Figure~\ref{fig:lower_fsm}.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\linewidth]{Figs/lowerframe.png}
    \caption{FSM logic integrated into \texttt{processLowerFrame()}.}
    \label{fig:lower_fsm}
\end{figure}

\textbf{Implementation Details:}
\begin{enumerate}
    \item \textbf{Emergency Frame Detection:} A voice-priority frame (QoS TID\,6, mapped to AC\_VO) arriving while the \texttt{carCrash} flag is set is treated as a collision sentinel. If the FSM is not already suppressed, the method immediately calls \texttt{triggerBlocking()}.
    \item \textbf{The \texttt{triggerBlocking()} Transition:} This routine enforces the BLOCKED state:
    \begin{itemize}
        \item Sets \texttt{blockingEndTime} to the current time plus the configured \newline \texttt{blockDuration}.
        \item Forces the FSM into \texttt{BLOCKED} via \texttt{switchState()}.
        \item Cancels ongoing EDCA timers and clears the current channel owner, freezing every EDCAF and imposing a silent guard interval.
    \end{itemize}
    This mechanism quenches collision cascades typical of hidden-terminal failures in dense platoons.
    \item \textbf{Post-Blocking Recovery:} When \texttt{blockingEndTime} expires, the controller returns to \texttt{LISTENING} if the quiet window is still valid; otherwise, it re-enters \texttt{ACTIVE} and resumes the periodic cycle.
    \item \textbf{Standard Frame Processing:} Non-emergency frames continue under standard EDCA rules but remain subject to FSM gating. Frame sequencing, acknowledgments, and retransmissions are preserved, though suspended whenever the machine is in \texttt{BLOCKED}.
\end{enumerate}

This design ensures a deterministic, reactive suppression mechanism: emergency events immediately pre-empt ACTIVE and LISTENING, guaranteeing a collision-free interval without relying on explicit preemption protocols.

The integration of the \texttt{EdcaFsmController} into the HCF/EDCA architecture is shown in Figure~\ref{fig:model_fsm}. The diagram synthesises the revised class structure and exposes the controller’s role in orchestrating channel access.

\textbf{Implementation Details:}
\begin{enumerate}
    \item \textbf{Controller as Submodule:} The \texttt{EdcaFsmController} is instantiated as a direct subcomponent of the \texttt{Hcf} class. It encapsulates the three-state machine (\texttt{ACTIVE}, \texttt{LISTENING}, \texttt{BLOCKED}), together with timers and state variables (\texttt{EdcaFsmState}) that regulate transitions.
    \item \textbf{Feature Switches:} The boolean flags \texttt{BEAperiodic} and \texttt{carCrash} are implemented at the controller level. These switches alter contention behaviour by silencing AC\_BE during critical windows or invoking BLOCKED preemption in response to emergency events.
    \item \textbf{Control Flow Hooks:} The original \texttt{processUpperFrame()} and \texttt{processLowerFrame()} methods are extended to trigger FSM events. Their interaction with the controller drives the finite-state transitions detailed in Figures~\ref{fig:upper_fsm} and~\ref{fig:lower_fsm}.
\end{enumerate}

This architecture enforces deterministic QoS at the MAC layer through behavioural gating rather than intrusive rewrites. Together with the upper- and lower-frame FSMs (Figures~\ref{fig:upper_fsm} and~\ref{fig:lower_fsm}), the integration establishes the timing semantics, bandwidth-reservation strategy, and fail-safe collision handling that underpin the DQ-WiFi enhancement.



\chapter{Preliminary Results}
\label{chap:results}

To assess how each incremental MAC feature influences network performance, we run repeatable packet-level simulations and record aggregate throughput, end-to-end delay, maximum delay and jitter for the relevant traffic classes. Results are reported as generic, untuned values so that improvements stem solely from protocol changes rather than parameter optimisation.

\textbf{BEAperiodic = polling-like service on \textbf{AC\_BE}.}
\texttt{BEAperiodic} is a boolean flag that turns the EDCA Best-Effort queue (AC\_BE) into a polling-style aperiodic server with a fixed duty cycle. When \texttt{BEAperiodic} is \emph{true}, the MAC FSM enforces
an \emph{ACTIVE}/\emph{LISTENING} schedule of period \(T=\texttt{activePeriod}+\texttt{listenPeriod}\):
(i) during \emph{ACTIVE}, AC\_BE is muted (no transmissions); (ii) during \emph{LISTENING}, AC\_BE is preferentially serviced and drains its backlog. This emulates a polling server that provides deterministic service opportunities to aperiodic traffic. In our mapping, \textbf{CONTROL} packets use AC\_BE; therefore their worst-case waiting time is upper-bounded by \(\texttt{activePeriod}\) (before the next LISTENING window), plus the in-window queuing time. We expose two switches in experiments: a global controller flag (\texttt{BEAperiodic=true}) and an EDCAF-level tag that marks AC\_BE as aperiodic.

\textbf{Common setup.} One 802.11 AP and \(N\) vehicles (STAs) in LOS; EDCA queues with DSCP\(\rightarrow\)UP classification; MAC FSM provides \emph{ACTIVE}, \emph{LISTENING}, and \emph{BLOCKED}. Each vehicle runs three apps:
\begin{enumerate}
  \item \textbf{CONTROL} (periodic, medium priority) \(\rightarrow\) \emph{AC\_BE} (subject to \texttt{BEAperiodic} when enabled),
  \item \textbf{SENSOR} (periodic, high priority) \(\rightarrow\) \emph{AC\_VI},
  \item \textbf{COLLISION} (burst, alert) \(\rightarrow\) \emph{AC\_VO}; may trigger \emph{BLOCKED} via the collision sentinel.
\end{enumerate}
Unless stated, we use INET 802.11ac defaults; EDCAF capacity \(-1\); \(\texttt{activePeriod}=3\,\mathrm{s}\), \(\texttt{listenPeriod}=1\,\mathrm{s}\).

This section evaluates the performance of the Auto5G QoS model, analyzing its ability to provide deterministic latency for critical traffic while managing other data flows.

End-to-End Delay Analysis (\ref{results:end}) quantifies the latency and jitter trade-offs, demonstrating how the model guarantees ultra-low latency for alerts and bounded delays for best-effort traffic.

\subsection*{Analysis}
The effects below follow directly from the delay decomposition in Section~\ref{sec:analysis}
(\(d_{\text{queue}}\), \(d_{\text{cycle}}\), \(d_{\text{access}}\), \(\tau_{\text{MAC}}\), and blackout \(\Delta_{\text{block}}\)).

\begin{itemize}
    \item \textbf{NOQOS} shows the highest collision frame delays and jitter because all traffic shares one FIFO.
    \item \textbf{EDCA\_ONLY} isolates queues, cutting average CONTROL delay by $\sim$40\% and COLLISION delay by two orders of magnitude, but worst-case latency is unknown.
    \item \textbf{CRASH\_QOS} adds the \emph{BLOCKED} state. Alert traffic (AC\_VO) now arrives in sub-millisecond time with micro-second jitter because it bypasses \(d_{\text{cycle}}\) and minimizes \(d_{\text{access}}\); however, CONTROL delay rises since the blackout inserts \(\Delta_{\text{block}}\) for all other queues.
    \item \textbf{CRASH\_QOS2} activates bandwidth-reservation only. The 3\,s \emph{ACTIVE} period mutes AC\_BE, so CONTROL frames see a deterministic bound dominated by \(d_{\text{cycle}}\le T_A=3\,\mathrm{s}\); the measured worst case of 3.04\,s matches \(T_A\) plus a small \(d_{\text{access}}\) term.
\item \textbf{CRASH\_QOS3} — reservation and blocking coexist. If no collision occurs, the 3\,s reservation bound is respected for CONTROL (\(d_{\text{cycle}}\le T_A\)). When a collision occurs, the blackout pre-empts the reservation; with a single event (\(k=1\)) the worst-case bound from Section~\ref{sec:analysis} reduces to
\[
d_{\text{crit,wc}} \;\le\; d_{\text{queue}} + T_A + \Delta_{\text{block}} + \tau_{\text{MAC}} ,
\]
which matches the observed \(7.74\,\mathrm{s}\). ALERT traffic keeps sub-millisecond latency because blocking grants it immediate priority and removes \(d_{\text{cycle}}\).

\end{itemize}

These generic results confirm the design trade-off: reservation guarantees a fixed upper bound for best-effort queues (mapped here as CONTROL Frames), while collision blocking preserves ultra-low latency for alerts—at the cost of elongating CONTROL latency whenever both mechanisms overlap.
